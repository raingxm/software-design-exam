# 文法与自动机深度讲解

## 1. 这一节到底在讲什么？

**不是在教你写代码，而是在讲「编译器怎么判断一段程序写得对不对」。**

| 概念 | 人话 |
| :--- | :--- |
| **文法** | 语言的「语法规则说明书」 |
| **自动机** | 按规则检查/识别字符串的「机器」 |

两者**一一对应**：某种文法能描述的语言，可以用某种自动机来识别。

### 编译器里的两条线

```text
你写的代码
    ↓
词法分析：拆成 token（int、abc、+、;）
    → 工具：正规式 / 有限自动机（3 型文法）
    ↓
语法分析：检查结构对不对（赋值、表达式、括号）
    → 工具：上下文无关文法 / 下推自动机（2 型文法）
    ↓
语义分析：检查类型对不对
```

**读教材时，每个概念都问一句：「这是给词法用的，还是给语法用的？」**

## 2. 教材写法 vs 人话翻译

| 教材里怎么说 | 实际上在说什么 |
| :--- | :--- |
| 「形式文法 G 是一个四元组…」 | 就是一套**语法规则** |
| 「VN 是非终结符，VT 是终结符…」 | **VN** = 还能继续展开的符号（如 E、T）；**VT** = 最终出现的真实字符（如 `+`、`*`、`id`） |
| 「产生式 P：α → β」 | 一条**替换规则**：「看到 α，可以换成 β」 |
| 「L(G) 是文法 G 生成的语言」 | 按这套规则能造出的**所有合法字符串** |
| 「有限自动机 M 是五元组…」 | 一个**逐字符扫描、靠状态跳转**的识别器 |
| 「上下文无关文法」 | 规则左边**只有一个**非终结符，适合描述**嵌套结构** |
| 「下推自动机」 | 带**栈**的自动机，能记住括号嵌套层数 |

## 3. 乔姆斯基文法四级（能力等级表）

从弱到强记 **3 → 2 → 1 → 0**：

| 类型 | 名称 | 能描述什么 | 对应自动机 | 编译里用在哪 |
| :--- | :--- | :--- | :--- | :--- |
| **3 型** | 正规文法 | 简单模式（标识符、数字） | **有限自动机** | **词法分析** |
| **2 型** | 上下文无关文法 | 嵌套结构（表达式、语句） | **下推自动机** | **语法分析** |
| 1 型 | 上下文有关文法 | 更复杂（部分自然语言） | 线性有界自动机 | 软考几乎不考 |
| 0 型 | 无限制文法 | 最强，图灵完备 | 图灵机 | 软考几乎不考 |

> **🔥 考点提醒 (Q1)**：程序语言文法 → **2 型**；词法分析 → **3 型（正规式/有限自动机）**。

### 等价关系

- 正规式 ≡ NFA ≡ DFA（可相互转换，都用于词法分析）
- 上下文无关文法 ≡ 下推自动机（用于语法分析）

### 为什么词法用有限自动机、语法用下推自动机？

| 任务 | 例子 | 需要什么 |
| :--- | :--- | :--- |
| 词法 | 识别 `abc123`、`while` | 简单模式匹配，**不需要记忆嵌套** |
| 语法 | 识别 `(a + b) * c`、括号配对 | **嵌套结构**，需要栈记住「还有几层括号没闭合」 |

## 4. E、T、F 是什么？（核心难点）

**E、T、F 不是代码里的变量，而是文法里的「占位符名字」**（非终结符）。

| 符号 | 英文 | 中文 | 代表什么 |
| :--- | :--- | :--- | :--- |
| **E** | Expression | **表达式** | 一整段能算的东西，如 `a + b * c` |
| **T** | Term | **项** | 比 E 小一级，通常是「乘法那一层」，如 `b * c` |
| **F** | Factor | **因子** | 最小单位：单个变量/数字，或括号包起来的一整段 |
| **id** | identifier | **标识符** | 真正的变量名，如 `a`、`x` |

**关键**：`id` 是真实程序里的东西；`E、T、F` 是文法规则里描述结构的「中间名字」。

### 经典文法规则

```text
① E → E + T      （表达式 = 表达式 + 项）
② E → T          （表达式也可以就是一个项）
③ T → T * F      （项 = 项 × 因子）
④ T → F          （项也可以就是一个因子）
⑤ F → id         （因子 = 一个变量名）
⑥ F → ( E )      （因子 = 括号里的表达式）
```

读法：**箭头左边可以替换成右边**。

### 为什么分 E、T、F 三层？

为了表达**运算符优先级**：

```text
E 层  →  管 +  -
T 层  →  管 *  /
F 层  →  管单个 id 或 ( ... )
```

所以 `a + b * c` 的结构是：

```text
        +
       / \
      a   *
         / \
        b   c
```

- `+` 在 E 层 → 优先级低，最后算
- `*` 在 T 层 → 优先级高，先算

**E/T/F 的分层，就是在文法里写死「先乘除后加减」。**

### 换成具体名字

`id + id * id` 等价于 `a + b * c`：

| 文法符号 | 对应真实内容 |
| :--- | :--- |
| 第一个 id | a |
| 第二个 id | b |
| 第三个 id | c |
| E | 整句 `a + b * c` |
| 右边 T | `b * c` 这一整块 |
| 左边 T | `a` |
| 各 F | 单个 a、b、c |

## 5. 自顶向下（LL）：从 E 展开到 id

目标：证明 `id + id * id` 符合这套规则。

**思路**：从最大的 **E** 出发，不断用规则「展开」，直到全是 `id` 和运算符。

### 完整推导

| 步骤 | 当前串 | 用了哪条规则 | 在干什么 |
| :---: | :--- | :--- | :--- |
| 0 | **E** | 起点 | 从表达式开始 |
| 1 | E **+** T | ① E→E+T | 猜这是「加法表达式」 |
| 2 | T **+** T | ② E→T | 左边 E 先展开成 T |
| 3 | F **+** T | ④ T→F | 左边 T 展开成 F |
| 4 | **id** + T | ⑤ F→id | 第一个 id 出现 |
| 5 | id + T **\* ** F | ③ T→T*F | 右边是「乘法项」 |
| 6 | id + F * F | ④ T→F | 左边 T 展开成 F |
| 7 | id + **id** * F | ⑤ F→id | 第二个 id |
| 8 | id + id * **id** | ⑤ F→id | 第三个 id，完成 ✓ |

### 语法树

```text
        E
       /|\
      / | \
     E  +  T
     |     /|\
     T    T * F
     |    |   |
     F    F   id
     |    |
    id   id
```

**自顶向下 = 从树根 E 往下长叶子**。

- 典型方法：递归下降、预测分析
- 记忆：**自上而下 = 自顶向下 = LL**

---

## 6. 自底向上（LR）：从 id 归约到 E

**思路相反**：手里只有 `id + id * id`，看到能合并的模式就「归约」成更大的符号。

### 用栈模拟

```text
步骤  栈里有什么          看到什么    动作
────────────────────────────────────────────
1     空                 id         先收下 id
2     id                 +          id → F → T
3     T                  +          收下 +
4     T +                id         收下 id
5     T + id             *          id→F→T；不能急着 T+T→E
                                       （后面还有 *，要先处理乘法）
6     T + T              *          收下 *
7     T + T *            id         收下 id
8     T + T * id                      id→F
9     T + T * F                       T*F → T
10    T + T                            T+T → E
11    E                                完成 ✓
```

**自底向上 = 从叶子 id 往上合并成 E**。

### 为什么第 5 步不能马上把 T + T 合成 E？

因为真实程序是 `id + id * id`，`*` 还没处理。

若过早合成 E，会变成 `(id + id) * id`，**优先级就错了**。

LR 先让 `id * id` 在 T 层合并，再和左边的 T 做 `T + T → E`，自然得到 `id + (id * id)` —— **先乘后加**。

- 典型方法：移进-归约
- 记忆：**自下而上 = 自底向上 = LR**

---

## 7. LL vs LR 对比

| | 自顶向下 LL | 自底向上 LR |
| :--- | :--- | :--- |
| **从哪开始** | 从 **E**（根） | 从 **id + id * id**（叶子） |
| **做什么** | **展开**（E→E+T→…→id） | **归约**（id→F→T→…→E） |
| **比喻** | 树从根往叶长 | 叶子往根合 |
| **典型方法** | 递归下降、预测分析 | 移进-归约 |

## 8. 完整例子：从源程序到语法树

源程序：

```c
x = a + 1;
```

### 词法分析（3 型 / 有限自动机）

输出单词流：

```text
id(x)  =  id(a)  +  num(1)  ;
```

### 语法分析（2 型 / 下推自动机）

构建语法树：

```text
      =
     / \
    x   +
       / \
      a   1
```

若写成 `= a + 1;`（缺少左值），语法分析报错 —— 结构不合法。

4. `id * id + id` 的结构树长什么样？`*` 和 `+` 谁先结合？

4. 结构：

```text

     /|\
    T  +  T

  |   |   |

  |
 id
```
