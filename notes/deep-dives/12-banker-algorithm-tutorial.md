# 银行家算法详解

> 死锁避免的核心算法，软考常考计算题

---

## 一、为什么叫"银行家"？

类比银行贷款：
- 银行有固定资金（系统资源）
- 客户申请贷款额度（进程声明最大需求）
- 银行每次放款前要确保：**即使所有客户都来提满额度，也能周转过来**
- 如果某次放款会导致资金链断裂 → 拒绝

**核心思想**：分配前先"模拟"，确认不会死锁才真正分配。

---

## 二、关键数据结构

假设有 **n 个进程**，**m 种资源**：

| 名称 | 含义 | 维度 |
|------|------|------|
| **Available** | 当前系统剩余的各类资源 | 1×m 向量 |
| **Max** | 每个进程的最大需求 | n×m 矩阵 |
| **Allocation** | 每个进程已分配的资源 | n×m 矩阵 |
| **Need** | 每个进程还需要多少 | n×m 矩阵 |

**关键公式**：

```text
Need[i] = Max[i] - Allocation[i]
```

---

## 三、安全性检查算法

判断当前状态是否**安全**（存在安全序列）：

```text
1. 初始化：
   Work = Available          // 可用资源的工作副本
   Finish[i] = false         // 标记进程是否完成

2. 找一个满足条件的进程 Pi：
   Finish[i] == false  且  Need[i] ≤ Work
   
   意思：Pi 还没完成，且剩余资源够它跑完

3. 如果找到：
   Work = Work + Allocation[i]   // Pi 跑完，回收资源
   Finish[i] = true
   回到第 2 步

4. 如果找不到：
   检查是否所有 Finish[i] == true
   - 全 true → 【安全】，找到的顺序就是安全序列
   - 有 false → 【不安全】
```

## 四、手算例题

**题目**：3 种资源 A、B、C，初始 Available = (3, 3, 2)

| 进程 | Max (A,B,C) | Allocation (A,B,C) |
|------|-------------|---------------------|
| P0 | 7, 5, 3 | 0, 1, 0 |
| P1 | 3, 2, 2 | 2, 0, 0 |
| P2 | 9, 0, 2 | 3, 0, 2 |
| P3 | 2, 2, 2 | 2, 1, 1 |
| P4 | 4, 3, 3 | 0, 0, 2 |

### 第一步：算 Need

```text
Need = Max - Allocation

P0: (7,5,3) - (0,1,0) = (7, 4, 3)
P1: (3,2,2) - (2,0,0) = (1, 2, 2)
P2: (9,0,2) - (3,0,2) = (6, 0, 0)
P3: (2,2,2) - (2,1,1) = (0, 1, 1)
P4: (4,3,3) - (0,0,2) = (4, 3, 1)
```

| 进程 | Need (A,B,C) |
|------|--------------|
| P0 | 7, 4, 3 |
| P1 | 1, 2, 2 |
| P2 | 6, 0, 0 |
| P3 | 0, 1, 1 |
| P4 | 4, 3, 1 |

### 第二步：找安全序列

```text
初始 Work = (3, 3, 2)

第 1 轮：找 Need ≤ Work 的进程
  P0: (7,4,3) ≤ (3,3,2)? 7>3 ✗
  P1: (1,2,2) ≤ (3,3,2)? ✓ → 选 P1
  
  Work = (3,3,2) + Allocation[P1] = (3,3,2) + (2,0,0) = (5, 3, 2)
  序列: [P1]

第 2 轮：Work = (5, 3, 2)
  P0: (7,4,3) ≤ (5,3,2)? 7>5 ✗
  P2: (6,0,0) ≤ (5,3,2)? 6>5 ✗
  P3: (0,1,1) ≤ (5,3,2)? ✓ → 选 P3
  
  Work = (5,3,2) + (2,1,1) = (7, 4, 3)
  序列: [P1, P3]

第 3 轮：Work = (7, 4, 3)
  P0: (7,4,3) ≤ (7,4,3)? ✓ → 选 P0
  
  Work = (7,4,3) + (0,1,0) = (7, 5, 3)
  序列: [P1, P3, P0]

第 4 轮：Work = (7, 5, 3)
  P2: (6,0,0) ≤ (7,5,3)? ✓ → 选 P2
  
  Work = (7,5,3) + (3,0,2) = (10, 5, 5)
  序列: [P1, P3, P0, P2]

第 5 轮：Work = (10, 5, 5)
  P4: (4,3,1) ≤ (10,5,5)? ✓ → 选 P4
  
  序列: [P1, P3, P0, P2, P4]
```

**结论**：存在安全序列 **P1 → P3 → P0 → P2 → P4**，当前状态**安全**。

## 五、资源请求算法

当进程 Pi 请求资源 Request[i] 时：

```text
1. 检查 Request[i] ≤ Need[i]
   否则出错（请求超过声明的最大值）

2. 检查 Request[i] ≤ Available
   否则 Pi 必须等待（资源不够）

3. 试探性分配：
   Available = Available - Request[i]
   Allocation[i] = Allocation[i] + Request[i]
   Need[i] = Need[i] - Request[i]

4. 执行安全性检查
   - 安全 → 正式分配
   - 不安全 → 回滚，Pi 等待
```

## 六、考试常见问法

| 问法 | 答题要点 |
|------|----------|
| 判断当前是否安全 | 算 Need，跑安全性检查，给出安全序列或说明不安全 |
| 能否满足某请求 | 先试探分配，再判安全性 |
| 找出所有安全序列 | 每轮可能有多个可选进程，穷举 |

---

## 七、一句话记忆

> **银行家 = 分配前先模拟，确认能"周转"才放款**
> 
> 安全序列 = 按某顺序执行，每个进程都能拿够资源跑完

## 八、自测题

**题目**：在上述例题基础上，P1 请求 (1, 0, 2)，能否分配？

<details>
<summary>点击查看答案</summary>

1. 检查 Request ≤ Need[P1]：(1,0,2) ≤ (1,2,2)? ✓
2. 检查 Request ≤ Available：(1,0,2) ≤ (3,3,2)? ✓
3. 试探分配：
   - Available = (3,3,2) - (1,0,2) = (2, 3, 0)
   - Allocation[P1] = (2,0,0) + (1,0,2) = (3, 0, 2)
   - Need[P1] = (1,2,2) - (1,0,2) = (0, 2, 0)
4. 安全性检查：Work = (2, 3, 0)
   - P1: (0,2,0) ≤ (2,3,0)? ✓ → Work = (5,3,2)
   - P3: (0,1,1) ≤ (5,3,2)? ✓ → Work = (7,4,3)
   - P0, P2, P4 依次可完成
   
**结论**：安全，可以分配。
