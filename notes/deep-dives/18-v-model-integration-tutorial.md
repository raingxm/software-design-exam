## 一、两张对照总表（先背这个）

### 1. V 模型：开发阶段 ↔ 测试级别

| 左侧（开发） | 右侧（测试） | 在验证什么 | 粒度 |
|--------------|--------------|------------|------|
| 需求分析 | **验收测试** | 是不是用户要的 | 用户/业务 |
| **概要设计** | **系统测试** | 整系统是否符合规格/架构目标 | **系统** |
| 详细设计 | **集成测试** | 模块拼装、接口是否正确 | 模块间 |
| 编码 | **单元测试** | 函数/类内部逻辑 | 模块内 |

```text
需求 ───────────────→ 验收
  \                     /
概要 ─────────────→ 系统      ← 常考：「概要对应？」→ 系统
    \                 /
  详细 ─────────→ 集成
      \             /
    编码 ─────→ 单元
```

### 2. 集成策略：桩 vs 驱动

| | **自顶向下** | **自底向上** |
|--|--------------|--------------|
| 组装顺序 | 先上层/主控，再往下挂 | 先底层，再往上组 |
| 真实模块在 | **上** | **下** |
| 缺的是 | 下层 | 上层 |
| 假模块叫 | **桩（stub）** | **驱动（driver）** |
| 假模块干什么 | 被上层**调用**，返回固定值/空实现 | **调用**下层，造参数、看结果 |
| 好处 | 早看清控制流、主流程 | 早测核心算法/底层服务 |
| 口诀 | 缺脚 → 用桩撑 | 缺头 → 用驱动喊 |

```text
自顶向下                    自底向上

   [A 真实]                   [驱动·假]
  /         \                     |
[桩 B]     [桩 C]              [D 真实]
```

## 二、V 模型详解

### 2.1 它在考什么？

V 模型 = 瀑布的变体：左边往下做设计，右边往上做验证；**同一高度左右对应**。  
题干若强调「开发与测试阶段一一对应 / 验证与确认」，优先想 V。

### 2.2 为什么「概要 ↔ 系统」，不是「概要 ↔ 集成」？

| 阶段产物 | 关注点 | 对应测试为何匹配 |
|----------|--------|------------------|
| **概要设计** | 子系统怎么切、主要接口、整体结构 | **系统测试**：整系统对照需求/规格，粒度同为「系统级」 |
| **详细设计** | 模块内部结构、模块间调用细节 | **集成测试**：模块组装与接口，粒度同为「模块间」 |
| **编码** | 具体语句 | **单元测试**：模块内部 |
| **需求** | 用户要什么 | **验收测试**：用户是否接受 |

**记忆锚点**：概要设计画的是「系统长什么样」→ 右边用**系统测试**验收这张图。

### 2.3 四对记忆卡（防混）

| 题干问 | 答 | 易错成 |
|--------|-----|--------|
| 概要设计对应？ | **系统测试** | 集成（错把概要当详细） |
| 详细设计对应？ | **集成测试** | 系统 |
| 需求分析对应？ | **验收测试** | 系统 |
| 编码对应？ | **单元测试** | 集成 |

### 2.4 和测试级别链的关系

测试执行顺序仍是：

```text
单元 → 集成 → 系统 → 验收
```

V 模型说的是「**依据哪份设计文档来设计该类测试**」，不是说「概要做完立刻做系统测试」。  
编码后先单元，再按详细设计做集成，再按概要/需求做系统，最后验收。

## 三、集成策略详解

### 3.1 为什么需要桩 / 驱动？

集成测的是**接口与组装**。现实中总有一层还没写完，就要先测已完成的那一层 → 用**假模块**顶替缺失方。

| 名称 | 英文 | 角色一句话 |
|------|------|------------|
| **桩** | stub | 「假下属」：接口在，内部空/固定返回，**被调用** |
| **驱动** | driver | 「假上司」：小测试程序，**主动调用**真实模块 |

### 3.2 对比：谁调用谁？

```text
顶向下：真实上层  --调用-->  桩（假下层）
底向上：驱动（假上层）  --调用-->  真实下层
```

判断口诀：

1. 看**先测哪一层**（上还是下）  
2. 假模块补在**对面那一层**  
3. 假的在下叫桩；假的在上叫驱动  

### 3.3 和其他策略的对比

| 策略 | 需要 | 特点 | 考试态度 |
|------|------|------|----------|
| 自顶向下 | **桩** | 早见控制结构 | 常考「需要什么」 |
| 自底向上 | **驱动** | 早测底层 | 常考「需要什么」 |
| 三明治/混合 | 桩 + 驱动 | 两边夹 | 知道名字即可 |
| 一次性组装（大爆炸） | — | 全写完再拼，接口问题难定位 | **反面** |

### 3.4 经典选择题怎么拆

- 「自顶向下集成通常需要？」→ **桩**  
- 「自底向上集成通常需要？」→ **驱动**  
- 「为未完成的下层模块提供接口」→ **桩**  
- 「编写主程序调用已完成的底层模块」→ **驱动**

## 四、自测（闭卷）

1. 概要设计对应的测试级别是？  
2. 详细设计对应的是？  
3. 自顶向下缺的是上层还是下层？用桩还是驱动？  
4. 用一句话区分桩和驱动（谁调用谁）。

<details>
<summary>答案</summary>

1. **系统测试**  
2. **集成测试**  
3. 缺**下层** → **桩**  
4. 桩被上层调用（假下属）；驱动调用下层（假上司）

</details>

## 五、口令卡（30 秒）

```text
V：需求↔验收 · 概要↔系统 · 详细↔集成 · 编码↔单元
集成：顶向下→桩（假下属）· 底向上→驱动（假上司）
锚点：概要=系统级结构 → 系统测试；详细=模块怎么接 → 集成测试
```
